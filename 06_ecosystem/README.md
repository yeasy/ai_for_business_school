# 第六章 平台与生态：开源框架与商业平台

前面几章讲清了智能体"是什么、怎么工作、靠哪些技术"。本章回答紧随其后的现实问题：真要动手，用什么造、去哪儿买、放在哪里跑、怎么验收。这些看似是技术团队与采购部门的日常，却是管理者最容易因为"看不懂"而放权过度的环节——而恰恰在这里，数据归属、供应商锁定、合规风险这些董事会级的问题被一笔一笔写进了合同。

本章按决策顺序展开五节。6.1 梳理开源框架图谱：LangChain/LangGraph、CrewAI、Microsoft Agent Framework、Dify、OpenAI Agents SDK 各自的定位与适用团队；6.2 看商业平台格局：国际四大阵营与国内主要平台，以及"跟着现有系统栈走"的选择逻辑；6.3 讲选型与谈判：供应商评估五问、合同三底线与 POC 验收要点；6.4 讲部署与数据安全：用"数据分级—部署匹配"框架回答董事会最常问的"数据放出去安不安全"；6.5 讲评估与运维：概率性系统为什么需要一条全新的质检线。

三处分工边界需要先说明。其一，6.3 只讨论供给侧的"怎么挑、怎么谈"；某个场景究竟该自建还是采购，属于需求侧的战略决策，归 [10.4 的 2×2 框架](../10_strategy/10.4_decision_matrix.md)。其二，6.4 只讨论数据分级与部署方式的匹配；监管条文与合规日程的全景，归 [12.2 监管地图](../12_governance/12.2_regulation.md)。其三，6.5 是单个智能体的技术质检；项目级的信任爬坡归 [9.5](../09_landing/9.5_trust_control.md)，企业级治理制度归 [12.3](../12_governance/12.3_governance_system.md)。

最后一句提醒：本章是全书时效性最强的一章，产品与版本信息核实至 2026 年年中，读到本章时具体格局可能又有变化。但"分级、匹配、验收、留权"这套决策方法不会过期——变的是货架上的商品，不变的是挑商品的尺子。
