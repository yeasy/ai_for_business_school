# 第四章 大模型的能力与局限

从本章起，全书进入原理篇：认知篇的商业判断背后，技术机理是什么，管理者要懂到什么深度，才能与技术团队平等对话。本章聚焦智能体的底座——大模型本身，依次回答四个问题。[4.1](4.1_next_token.md) 回答"它内部在做什么"：从"预测下一个词"这个简单机制出发，解释训练过程、规模定律与涌现能力，以及 2026 年"推理成为标配"的现状。[4.2](4.2_delegation.md) 回答"怎么与它高效协作"：提示词技巧已在贬值，管理者真正需要长期持有的是任务定义、上下文供给、验收标准三件事。[4.3](4.3_hallucination.md) 回答"它为什么会出错"：拆解幻觉与知识截止的技术成因，说明这类错误为什么不可根除、但完全可以管理。[4.4](4.4_model_choice.md) 回答"企业该怎么选"：能力、成本、延迟的三角权衡，闭源 API 与开放权重两条路线，以及避免供应商锁定的多模型策略。

分工边界需要交代清楚。其一，本章与 [1.2](../01_essence/1.2_llm_base.md) 的分工是"技术机理对经济属性"：1.2 论证大模型是新的通用目的技术、其经济含义是什么，本章解释它在机器层面如何做到、又在哪里失灵，两处不重复。其二，全书四次讨论"边界"，各有分工：[1.3](../01_essence/1.3_boundary.md) 讲商业层现象（能做什么与不能做什么），[2.4](../02_agent/2.4_scenarios.md) 讲任务判据（流程可理清、结果可核验），本章 4.3 讲技术成因（为什么会错），[7.5](../07_value/7.5_real_vs_fake.md) 讲经济判据（高频、结构化、可审计）。其三，本章保持"管理者够用"的深度：全章没有一个数学公式，凡属工程细节（推理模型、RAG、工具调用、上下文工程）只给结论与链接，展开在第五章。

读完本章，面对"AI 为什么会说错话""该买哪家模型"这类问题，管理者应当能给出有技术依据的商业判断，而不是把问题原样转发给技术部门。
